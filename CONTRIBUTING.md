## Code of Conduct

Be respectful. Be constructive. Assume good intent.

Harassment, discrimination, or hostile behavior is not tolerated. Disagreements are fine; personal attacks are not. Maintainers may remove or reject contributions that violate this principle.


## Ways to Contribute

You can contribute by:

* Fixing bugs
* Implementing new features
* Improving documentation
* Refactoring or cleaning up code
* Improving build scripts or tooling
* Reporting bugs or regressions
* Reviewing pull requests


## Getting Started

1. Fork the repository
2. Make your changes
3. Test your changes where applicable
4. Open a pull request one the `dev` branch


## Development Guidelines

* Keep changes focused. One pull request = one logical change.
* Avoid unnecessary rewrites or style-only changes unless discussed first.
* Prefer clarity over cleverness.
* If you introduce new behavior, update documentation accordingly.

If your change is large or architectural, consider opening an issue first to discuss it.


## Commit Messages

Write clear, descriptive commit messages.

**Good examples:**

* `Handle null pointer in scheduler`
* `Added basic framebuffer driver`
* `Clarified build instructions`

**Avoid:**

* `fix stuff`
* `updates`
* `lol`


## Pull Requests

When opening a pull request:

* Clearly describe *what* the change does
* Explain *why* the change is needed
* Reference relevant issues if applicable
* Mention any known limitations or follow-up work

Pull requests may be reviewed, requested to change, or rejected. This is not personal.


## Reporting Bugs

If you find a bug, please open an issue and include:

* What you expected to happen
* What actually happened
* Steps to reproduce (if possible)
* Logs, screenshots, or crash output (if available)
* Your environment (architecture, build options, etc.)

Clear bug reports save time for everyone.


## Suggesting Features

Feature requests are welcome.

Please include:

* The problem you’re trying to solve
* Why this feature belongs in ErisOS
* Any prior art or references

Not all feature requests will be accepted. ErisOS has a specific scope and philosophy.


## Style & Philosophy

ErisOS values:

* Simplicity over complexity
* Explicit behavior over implied actions
* Learning-friendly code

If you’re unsure whether something fits, ask first.


## Final Notes

By contributing, you agree that your contributions may be used, modified, and redistributed as part of the ErisOS project under the MIT license.
